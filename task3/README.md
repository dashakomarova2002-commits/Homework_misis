# Task3 — Онтология «Автономные мобильные роботы»: классификация рисковых роботов (Protege + SWRL)

## Краткое описание

Создана русскоязычная онтология предметной области автономных мобильных роботов, предназначенная для классификации роботов на основе используемых маршрутов движения и информации о критических дефектах.  
Онтология описывает роботов, маршруты движения, роботизированные системы, оборудование и дефекты, а также включает **транзитивное** свойство для включения компонентов в систему и **функциональные** свойства для задания однозначного маршрута и критического дефекта робота.

Поверх онтологии реализовано SWRL-правило импликации, которое средствами логического вычислителя (Drools + reasoner) автоматически относит роботов к классу `РисковыйРобот` при выполнении заданных условий.  
Онтология разработана в **Protege 4.5.29** и экспортирована в **Turtle (.ttl)**.

## Критерии

- **Базовая предметная область:** эксплуатация автономных мобильных роботов и их техническое состояние.
- **Цель классификации:** отнесение робота к классу `РисковыйРобот` при наличии критичного оборудования в маршруте и критичного дефекта.
- **Онтология отвечает на вопрос:** «Какие роботы являются рисковыми с точки зрения их маршрутов движения и критических дефектов?». То есть для любого конкретного робота она позволяет автоматически определить, относится ли он к классу `РисковыйРобот` при заданном маршруте (с критичным оборудованием) и наличии критического дефекта.
- **Типы отношений (≥2):**
  - Объектные свойства: `использует_маршрут`, `включает_компонент`, `имеет_критичный_дефект`.
  - Используются характеристики **Functional** и **Transitive**.
- **Сущности (для проверки):** `Робот_TurtleBot`, `Робот_CleanerBot`, `Маршрут_Складской`, `СистемаНавигации`, `Лидар_RPLidar`, `Дефект_ОтказЛидара`, `Дефект_ПовреждениеКорпуса`.

## Структура онтологии

### Классы

- `Робот`
- `МаршрутДвижения`
- `РоботизированнаяСистема`
- `СенсорнаяСистема ⊑ РоботизированнаяСистема`
- `Оборудование ⊑ РоботизированнаяСистема`
- `КритичноеОборудование ⊑ Оборудование`
- `Дефект`
- `КритичныйДефект ⊑ Дефект`
- `РисковыйРобот ⊑ Робот`

### Объектные свойства

- `использует_маршрут : Робот → МаршрутДвижения` — **Functional**  
  (каждый робот имеет один основной маршрут движения).

- `включает_компонент : РоботизированнаяСистема → РоботизированнаяСистема` — **Transitive**  
  (маршрут включает системы, системы включают оборудование; транзитивность позволяет выводить, что маршрут включает оборудование).

- `имеет_критичный_дефект : Робот → Дефект` — **Functional**  
  (у робота фиксируется один критический дефект для целей классификации).

### Классовые ограничения (аксиомы)

- `СенсорнаяСистема ⊑ РоботизированнаяСистема`
- `Оборудование ⊑ РоботизированнаяСистема`
- `КритичноеОборудование ⊑ Оборудование`
- `КритичныйДефект ⊑ Дефект`
- `РисковыйРобот ⊑ Робот`

### Индивиды (для проверки работы SWRL)

**Маршруты движения и системы:**

- `Маршрут_Складской : МаршрутДвижения`
- `СистемаНавигации : СенсорнаяСистема`
- `Лидар_RPLidar : Оборудование, КритичноеОборудование`

**Дефекты:**

- `Дефект_ОтказЛидара : Дефект, КритичныйДефект`
- `Дефект_ПовреждениеКорпуса : Дефект`

**Роботы:**

- `Робот_TurtleBot : Робот`
- `Робот_CleanerBot : Робот`

### Связи между индивидами

**Структура системы (транзитивная цепочка):**

- `Маршрут_Складской включает_компонент СистемаНавигации`
- `СистемаНавигации включает_компонент Лидар_RPLidar`

За счёт **Transitive** у `включает_компонент` reasoner выводит:  
`Маршрут_Складской включает_компонент Лидар_RPLidar`.

**Роботы и маршруты:**

- `Робот_TurtleBot использует_маршрут Маршрут_Складской`
- `Робот_CleanerBot использует_маршрут Маршрут_Складской`

**Роботы и дефекты:**

- `Робот_TurtleBot имеет_критичный_дефект Дефект_ОтказЛидара`
- `Робот_CleanerBot имеет_критичный_дефект Дефект_ПовреждениеКорпуса`

## SWRL-правило классификации

Для реализации содержательно значимой классификации введено правило на языке SWRL, которое определяет, что робот является рисковым, если:

- он использует маршрут, который (в транзитивном смысле) включает критичное оборудование;
- у робота есть критический дефект.

Правило в нотации SWRL:

```text
Робот(?x) ^ 
использует_маршрут(?x, ?r) ^ 
включает_компонент(?r, ?o) ^ 
КритичноеОборудование(?o) ^ 
имеет_критичный_дефект(?x, ?d) ^ 
КритичныйДефект(?d) 
-> РисковыйРобот(?x)
```

При выполнении этого правила reasoner автоматически относит индивид `Робот_TurtleBot` к классу `РисковыйРобот`, а `Робот_CleanerBot` — нет (из-за некритичного дефекта).

## Как было сделано в Protege (кратко)

1. **Classes:** созданы классы `Робот`, `МаршрутДвижения`, `РоботизированнаяСистема`, `СенсорнаяСистема`, `Оборудование`, `КритичноеОборудование`, `Дефект`, `КритичныйДефект`, `РисковыйРобот`; настроены отношения подклассов согласно структуре, указанной выше.
2. **Object properties:** созданы свойства:
   - `использует_маршрут (Робот → МаршрутДвижения)` с характеристикой **Functional**;
   - `включает_компонент (РоботизированнаяСистема → РоботизированнаяСистема)` с характеристикой **Transitive**;
   - `имеет_критичный_дефект (Робот → Дефект)` с характеристикой **Functional**.
3. **Individuals:** добавлены индивиды `Маршрут_Складской`, `СистемаНавигации`, `Лидар_RPLidar`, `Дефект_ОтказЛидара`, `Дефект_ПовреждениеКорпуса`, `Робот_TurtleBot`, `Робот_CleanerBot` с указанием типов и объектных связей через `включает_компонент`, `использует_маршрут`, `имеет_критичный_дефект`.
4. **Reasoner (базовая проверка):** запущен reasoner (HermiT / Pellet); проверено, что транзитивность `включает_компонент` даёт вывод `Маршрут_Складской включает_компонент Лидар_RPLidar`.
5. **SWRLTab:** включена вкладка SWRLTab, создано правило `Правило_РисковыйРобот` в виде импликации SWRL, как указано выше.
6. **Запуск SWRL:** выполнена трансляция OWL+SWRL → Drools, затем **Run Drools**, после чего reasoner синхронизирован.
7. **Проверка вывода:** во вкладке **Individuals** для `Робот_TurtleBot` в **Inferred Types** появился класс `РисковыйРобот`; для `Робот_CleanerBot` — нет.
8. **Экспорт:** онтология с SWRL-правилом экспортирована в формате **Turtle (.ttl)**.

## Практическое применение

Данная онтология может использоваться в системах мониторинга парка автономных роботов для:
- Автоматического выявления проблемных роботов
- Планирования технического обслуживания
- Оптимизации маршрутов движения
- Управления рисками в логистических системах
